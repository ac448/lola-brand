LOLA · Product Overview · Spec 02

How LOLA works

LOLA is an AI routing agent for creative and marketing teams. She reads incoming work requests, matches them against your team's own routing rules, and proposes an owner, a project, and a priority. A person approves every decision. She never assigns work on her own.

Request lifecycle

1

A request comes in

Anyone on the team submits a structured work request form inside the project management tool. No new software to learn, the intake lives where the work already lives.

2

LOLA reads and reasons

The request hits a webhook. LOLA sends the form contents to Claude via the Anthropic API alongside the team's routing rulebook, a plain-language document the team owns and can edit. She identifies the request type, the right owner, and the deadline risk.

3

She proposes, and shows her work

LOLA posts a proposal as a comment: suggested owner, project, priority, and the specific rule she used to get there, e.g. matched: rule 2.1 (event collateral → design queue). Every proposal is auditable because every proposal cites its source.

state → PROPOSED (warm chip)

?

No matching rule? She stops.

If a request doesn't map cleanly to the rulebook, LOLA doesn't guess. She flags it for the triage owner with a note about what was missing. Ambiguity is escalated, never absorbed.

state → NEEDS HUMAN (cream chip)

4

A human approves. Work moves.

The triage owner accepts, edits, or rejects the proposal in one click. On approval, the task is assigned and routed. Cool tones in the interface are earned: they only ever appear downstream of a human yes.

state → ROUTED (cool chip)

The stack

Intelligence

Anthropic API

Claude handles classification and routing logic against the team rulebook.

Runtime

Node.js

Lightweight service receiving webhooks and orchestrating the round trip.

Hosting

Vercel

Serverless deploy. No infrastructure for a creative team to babysit.

Workspace

Asana, today

First integration. The routing engine is tool-agnostic by design.

Design principles

Propose-only by default

LOLA ships in safety mode. Autonomy is something a team can grant later, rule by rule, once trust is earned. It is never the starting point.

The rulebook belongs to the team

Routing logic lives in a plain-language document the team edits, not in code. Changing how work flows is a writing task, not a ticket to engineering.

Every decision cites a rule

No black-box assignments. If LOLA can't point to the rule that produced a proposal, the proposal doesn't get made.

Color carries state

Warm means proposed. Cool means approved and routed. Cream means a human is needed. The interface tells the truth at a glance.
